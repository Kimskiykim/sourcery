# Development Guidelines

## Цель

Проект больше не ведётся как быстрый прототип. Все новые изменения должны писаться как production-grade код: с явными контрактами, тестами, ограниченной сложностью и понятной архитектурой.

## Основные принципы

### TDD

Правило по умолчанию: сначала тест, потом реализация.

Это особенно обязательно для:
- `src/core/storage/*`
- `src/core/workspace/*`
- HTTP API
- `src/core/wiki/*`
- save / rename / conflict flows

Новая функциональность без тестов допускается только для очень тонких UI-правок, которые не меняют доменную логику.

### KISS

Мы не строим абстракции "на всякий случай".

Правила:
- один модуль решает одну конкретную задачу
- не добавлять слои, если они не уменьшают связанность
- не вводить generic framework внутри проекта без явной необходимости
- не превращать compat layer в основной доменный слой

### SOLID

Используем практично, без догматизма.

Правила:
- single responsibility: storage, workspace, wiki, compat и UI не смешиваются
- dependency direction: UI зависит от core-контрактов, а не наоборот
- interface boundaries: внешние слои работают через явные типы и API
- compat API не является source of truth
- plugin host не должен напрямую управлять файловой системой

### Clean Code

Правила:
- маленькие файлы и маленькие функции
- явные типы на границах модулей
- минимум скрытого mutable state
- ошибки являются частью контракта
- ad hoc логика в UI недопустима, если это domain behavior
- naming должен отражать домен, а не случайную реализацию

## Engineering Rules

### 1. Ни одной новой фичи без тестов

Минимум:
- unit tests для core-логики
- integration tests для HTTP API, если меняется transport behavior

### 2. Source of truth

Источник истины для заметок:
- markdown-файлы в `vault/`

Если позже появится `SQLite`, он должен быть только индексом или кэшем, а не primary storage.

### 3. Слои системы

Допустимая модель зависимостей:

- `core/storage`
- `core/workspace`
- `core/wiki`
- `compat/obsidian`
- `app.ts` / UI
- `server.ts` / transport

Правила:
- `workspace` может использовать `storage`
- `wiki` может использовать `workspace`
- `compat` может использовать `workspace` и `wiki`
- UI не должен знать детали файлового хранения
- transport не должен содержать domain logic

### 4. Save guarantees

Любые изменения в note lifecycle должны учитывать:
- flush перед закрытием/переключением
- предсказуемое поведение rename
- отсутствие silent data loss
- явный статус сохранения
- тесты на конфликт имён и персистентность

### 5. Error model

Нельзя оставлять ошибки как случайный набор `throw`.

Требования:
- у доменных ошибок должен быть понятный контракт
- HTTP ошибки должны маппиться детерминированно
- UI должен показывать ожидаемые ошибки предсказуемо

### 6. Obsidian-like compatibility

Мы делаем не клон внутренних реализаций Obsidian, а совместимый по духу API-слой.

Правила:
- core остаётся нашим
- compat layer повторяет только полезный surface API
- нельзя строить core вокруг случайных ограничений compat-слоя
- полная plugin compatibility не обещается без явного подтверждения

### 7. Agent Write Policy

На текущем этапе Sourcery не является authority по filesystem permissions для внешнего агента.

Правила:
- source of truth для knowledge-layer остаются markdown-файлы в подключённом `vault` или `notesRoot`
- если агент уже имеет runtime-доступ к разрешённой markdown-папке, он может менять `.md` напрямую
- Sourcery должен корректно подхватывать такие внешние изменения через watcher/revision-механизм
- write scope агента должен ограничиваться sandbox/runtime permissions, а не только UI или prompt-инструкциями
- отдельная permission system для внешних write-доступов вне локально разрешённой markdown-папки пока не вводится

## Definition of Done

Изменение считается завершённым, если:
- поведение покрыто тестами
- архитектурный слой выбран правильно
- код собирается без ошибок
- нет очевидной утечки ответственности между слоями
- документация обновлена, если изменился контракт или правила разработки

Для UI-изменений, которые затрагивают загрузку приложения, навигацию между заметками,
виды editor/preview/graph/memory или save status, нужно запускать:

```bash
npm run smoke:ui
```

Для изменений agent API, MCP, workspace connections или context pack нужно запускать:

```bash
npm run smoke:agent
```

Перед завершением milestone-level изменений запускать:

```bash
npm run verify
```

## Запрещённые практики

- добавлять фичи поверх нестабильного save flow без тестов
- смешивать UI-код и domain logic в одном месте
- тащить SQL как primary storage
- копировать Obsidian API без понимания, зачем именно нужен конкретный surface
- усложнять архитектуру ради "гибкости на будущее"

## Ближайший режим работы

С этого момента приоритет такой:

1. тестовая инфраструктура
2. тесты для `core/storage`
3. тесты для `core/workspace`
4. интеграционные тесты API
5. только потом новые функции
